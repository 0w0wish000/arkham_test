# 03 · 技術需求盤點

回答你的三個核心評估問題:**核心技術需求**、**能不能用 Java**、**連線架構(LAN)** 與 **是否於 server 存放資料**。

## 2. Java 可行性評估 ✅(能,而且是強項)

### 2.1 直接前例(業界已證明)
「Java + 主從式 + 完整規則引擎 + 數千張卡」不是理論——已有兩套成熟的 MtG 引擎做到:

| 專案 | 語言 | 架構 | 客戶端 UI | 規模 | 佐證 |
|---|---|---|---|---|---|
| **XMage** | Java | 主從式(server + client) | Java(Swing) | 兩萬張以上卡、幾乎每張一個 Java 類別 | 完整規則自動結算、線上/區網對戰 |
| **Forge** | Java | 單機 + 連線 | Java(**LibGDX**,跨桌機/Android) | 兩萬張以上卡、內建 AI | 資料/腳本驅動的卡牌效果 |

> AHLCG 的卡池與規則複雜度**遠小於** MtG。既然 Java 能扛 MtG 等級,扛 AHLCG **綽綽有餘**。(此段的專案事實與版本細節,由背景研究查證後補上引用來源。)

### 2.2 Java 的優勢與適配
- **規則引擎(T1/T2)**:強型別 + OOP + 成熟工具鏈,非常適合複雜規則建模與大型測試套件。
- **後端(T8)**:**Spring Boot** 生態極成熟(REST、安全、ORM/JPA、驗證)。本專案有可用的 Spring Boot 開發技能可加速。
- **連線(T3)**:Java NIO / **Netty** / Jakarta WebSocket 都是一流選擇;turn-based 低頻同步對 Java 完全無壓力。
- **跨平台**:JVM 一次寫、多平台跑(Win/macOS/Linux)。
- **確定性/可測試**:headless 引擎在 JVM 上易於單元/情境測試 + CI。

### 2.3 客戶端技術選項(可與 Java 引擎解耦)
因為架構是「引擎在 server、客戶端只呈現」,**客戶端技術可獨立選擇**:

| 選項 | 優點 | 缺點 | 建議 |
|---|---|---|---|
| **Java + JavaFX** | 純 Java 全端;桌面卡牌 UI 夠用;CSS 樣式/動畫 | 手機/Web 較弱;動畫表現力中等 | LAN MVP、桌面優先可選 |
| **Java + LibGDX** | 跨桌機 + **Android/iOS**;遊戲級渲染(如 Forge) | 學習曲線;偏低階 | 想要遊戲感 + 未來手機版 |
| **Web(TypeScript/React/Phaser)** | 跨平台最廣、免安裝、易更新 | 需另一套語言;與 Java 引擎經 WebSocket 溝通 | 商業版觸及面最大 |
| **Unity / Godot** | 動畫/特效最強、商業卡牌遊戲常用 | 需 C#/GDScript;整合成本 | 追求高品質商業視覺時 |

> **建議路線(2026-07 拍板):** **後端/引擎鎖定 Java**(無爭議);**客戶端採「網頁技術」** —— TypeScript + **PixiJS**(2D 渲染)+ HTML/CSS 選單,經 **WebSocket** 接 Java 引擎;桌面版 / Steam / LAN 用 **Tauri**(比 Electron 輕)打包內嵌。理由:商業 + 廣觸及 + 頻繁更新 + 卡牌屬輕渲染。**替代方案:** 若團隊堅持純 Java 單一語言、以 Steam 離線為主,改用 **LibGDX**(Forge 前例)。
>
> ✅ **已有可運作的網頁原型:** `prototype/index.html` — 展示「圖版分佈流程 + 戰鬥流程 + 8 步技能檢定 + 混沌袋」,已用瀏覽器實測通過。此原型同時是客戶端的「畫面規格」活文件。

### 2.4 結論
> **能用 Java 嗎?→ 能,且建議後端與規則引擎就用 Java。** 唯一需要抉擇的是「客戶端要不要也用 Java」;若你團隊是 Java 背景、又想快,LibGDX/JavaFX 可全端 Java 落地。

## 3. 連線架構(LAN → 雲端)

### 3.1 模型:權威伺服器(唯一安全解)
- 客戶端送 **意圖(intent)/選擇(choice)**;伺服器**驗證 + 結算 + 廣播**過濾後的狀態。
- 合作遊戲雖無 PvP 作弊誘因,但**規則一致性、隱藏資訊(遭遇牌堆/混沌袋)** 決定了必須權威伺服器。

### 3.2 LAN 初期作法
- 某位玩家的客戶端**內嵌 Game Server**(host 模式),監聽 LAN 埠。
- 其他玩家輸入 **主機 IP:埠** 加入(或用區網探索 UDP broadcast/mDNS 自動找房)。
- 傳輸:**WebSocket over TCP**(可靠、有序;turn-based 不需 UDP 即時性)。LAN 內延遲可忽略。
- **好處:同一套 Game Server 程式,LAN 時內嵌、雲端時獨立部署,不需兩套。**

### 3.3 通訊協定
- **傳輸**:WebSocket(LAN 與雲端通用;未來 Web 客戶端可直接用)。
- **訊息類型**:`Intent`(玩家動作)、`ChoiceResponse`(回應決策)、`StateSync`(過濾後狀態/差異)、`EventLog`(動作播報)、`Chat`。
- **序列化**:turn-based 低頻 → **JSON 即可**(易除錯、易跨語言);要更省頻寬再換 Protobuf/MessagePack。
- **同步策略**:伺服器持有完整狀態 → 送每客戶端「過濾視圖 + 事件流」;可做全量快照(狀態小)或增量差異。
- **韌性**:斷線重連(伺服器保留 session,客戶端重取快照);host 遷移(LAN 若 host 離線)為進階議題。

### 3.4 從 LAN 到雲端要補什麼
房號/邀請碼、配對、NAT 穿透或中繼(雲端伺服器直接當中繼即免穿透)、負載與房間編排(Kubernetes/遊戲伺服器編排)。**因架構相同,主要是「加維運與規模化」而非「重寫」。**

## 4. 是否於 Server 存放資料?📦(分類回答)

**簡答:** 靜態內容可彈性;**動態玩家資料在商業版必須放 server**;純 LAN 自用可本機,但**戰役存檔即使 LAN 也建議共享儲存**。

### 4.1 資料分類與儲存建議
| 資料 | 純 LAN 自用 | 商業雲端 | 理由 |
|---|---|---|---|
| **靜態卡表/劇本/規則資料** | 隨客戶端打包 | **內容伺服器/CDN 版本化派送** | 商業版要能不重裝就上新卡、控管擁有內容 |
| **卡牌圖像** | 隨客戶端 或 本機快取 | **物件儲存 + CDN** | 大檔、需快取;**版權屬 FFG**(見風險 R-01) |
| **帳號/身分** | 本機或免帳號 | **Server(必需)** | 商業營運核心 |
| **擁有內容(Entitlement)** | 不需(全開) | **Server(必需)** | 防盜版、決定卡池/劇本可用性 |
| **牌組(Deck)** | 本機檔案 | **Server(建議)** | 跨裝置、分享、與戰役綁定 |
| **戰役存檔(Campaign)** | **建議共享儲存**(見下) | **Server(必需)** | 多人共用一場長期戰役,任一人續開 |
| **對局狀態(進行中)** | 由 host 記憶體 + 本機存檔 | **Server 持久化** | 存檔/續玩/重連/重播 |
| **遙測/分析/客服** | 無 | **Server** | 營運需要 |

### 4.2 為什麼「戰役存檔」即使 LAN 也建議共享儲存
AHLCG 戰役是**跨數場、數週的長期進度**(XP、創傷、日誌、分支)。若只存於某位 host 的本機:
- 換人當 host 就讀不到進度;host 檔案遺失=整段戰役沒了;難以續玩。
- 折衷方案(LAN 無雲端時):每場結束把**戰役存檔匯出成檔案**由玩家保管/雲端硬碟共享,或架一台輕量區網存檔服務。

### 4.3 落地建議(儲存技術)
- **關聯資料**(帳號、擁有、牌組、戰役):**PostgreSQL**。
- **會話/配對/對局熱資料**:**Redis**。
- **卡牌圖像/大型靜態**:物件儲存(S3/GCS)+ **CDN**。
- **關鍵設計**:把持久化藏在 **Repository 介面** 後面 → LAN 版用「本機檔案/內嵌 SQLite」實作,雲端版換 PostgreSQL 實作,**遊戲邏輯不變**。這讓你「初期 LAN 本機、之後雲端」平滑升級。

### 4.4 結論
> **要不要放 server?** 商業目標下,**帳號/擁有內容/牌組/戰役/對局狀態最終都要 server 端**。但**初期 LAN MVP 可以先全部本機/host 端**,只要把儲存層抽象化(Repository 介面),日後搬雲不必改遊戲邏輯。**戰役存檔是唯一「即使 LAN 也建議儘早共享」的資料。**

## 5. 建議技術棧(Reference Stack)

| 層 | LAN MVP | 商業雲端 |
|---|---|---|
| 規則引擎 | **Java**(headless、確定性、大測試套件) | 同左 |
| 遊戲伺服器 | Java + **WebSocket**(內嵌於 host) | Java + Netty/WebSocket,獨立部署、可水平擴充 |
| 後端服務 | 內嵌精簡(或省略) | **Spring Boot**(REST + 安全 + JPA) |
| 客戶端 | **JavaFX 或 LibGDX**(全 Java) | 同左,或加 Web(TS)/Unity 客戶端 |
| 資料儲存 | 本機檔案 / 內嵌 SQLite | **PostgreSQL + Redis + 物件儲存/CDN** |
| 序列化/協定 | JSON over WebSocket | 同左(可選 Protobuf) |
| 測試/CI | JUnit 情境測試 + CI | 同左 + 端對端/負載測試 |
| 部署 | 玩家本機 | 容器化(Docker/K8s)、遊戲伺服器編排 |

## 6. 非功能性需求(Non-Functional)
- **可測試性**:引擎必須可無頭執行 + 情境化斷言(數千張卡的正確性命脈)。
- **確定性/可重播**:集中 seedable RNG;存檔即可重現。
- **可維護的內容管線**:新卡以資料/腳本加入,避免動核心。
- **i18n**:多語系(含官方繁中);文字與資料分離。
- **可觀測性**:對局事件日誌、錯誤回報(商業版)。
- **安全**:伺服器權威、輸入驗證、帳號/金流安全(商業版)。
