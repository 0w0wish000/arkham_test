# 01 · 規則與玩法整理(電子化建模視角)

> 目的:把 AHLCG 的規則抽象成「可被程式建模」的結構。每節末的 **🔧 建模重點** 是給規則引擎的提示。
>
> 本文為**玩法概觀**;**逐條規則、關鍵字定義、時機/優先權規格、卡牌解剖→資料欄位**見 [05-rules-engine-spec.md](05-rules-engine-spec.md)(直接取自官方規則書 `rulebook/ahc100_rulebook-web.pdf`,Revised Core Set)。

AHLCG 是 **1–4 人合作** 的敘事型卡牌遊戲(LCG, Living Card Game)。玩家各自扮演一位「調查員」,合力在一個個「劇本(Scenario)」中推進劇情、對抗由遊戲本身操控的「遭遇牌堆」。多個劇本串成一段「戰役(Campaign)」,調查員的成長、創傷與抉擇會跨劇本延續。

---

## 1. 遊戲組成

### 1.1 調查員(Investigator)
每位調查員是一張大卡,定義:
- **四項技能值**:意志 Willpower(意)、智力 Intellect(智)、戰鬥 Combat(戰)、敏捷 Agility(敏)。
- **生命值 Health** 與 **理智值 Sanity**(分別承受「傷害 Damage」與「恐懼 Horror」)。
- **職業(Class)**:守護者 Guardian、探求者 Seeker、流氓 Rogue、秘術師 Mystic、生存者 Survivor(另有中立 Neutral)。
- **特殊能力**(獨有的常駐/觸發能力)。
- **古老印記效果(Elder Sign, ⭐)**:抽到 ⭐ 混沌標記時,依此調查員定義結算。
- **牌組構築需求(Deckbuilding Requirements)**:可用職業/等級、牌組張數、必帶的「招牌卡(Signature)」與「招牌弱點(Signature Weakness)」。

> **修訂核心盒 5 位調查員(範例):** Daniela Reyes(Guardian 守護者)、Joe Diamond(Seeker 探求者)、Trish Scarborough(Rogue 流氓)、Dexter Drake(Mystic 秘術師)、Isabelle Barnes(Survivor 生存者)。各自有獨特能力、⭐ 效果、招牌卡與招牌弱點。

### 1.2 玩家卡(Player Cards)——組成牌組
| 類型 | 說明 | 關鍵屬性 |
|---|---|---|
| **支援 Asset** | 打到檯面長期存在(武器、盟友、法術、道具) | 費用、佔用欄位(Slot)、可能有生命/理智、技能圖示 |
| **事件 Event** | 一次性效果,打出後進棄牌堆 | 費用、觸發時機、技能圖示 |
| **技能 Skill** | 檢定時從手牌「投入(Commit)」,提供圖示加值 | 提供的技能圖示(可含 wild/百搭) |

- **欄位(Slots)**:手部 Hand(×2)、秘術 Arcane(×2)、盟友 Ally(×1)、身體 Body、配件 Accessory、塔羅 Tarot 等。同欄位有數量上限,超過需棄置。
- **等級(Level)0–5**:高等級卡需以「經驗值(XP)」購入。
- **弱點(Weakness)**:混入牌組的負面卡(基本弱點 + 調查員專屬弱點)。

### 1.3 遭遇卡(Encounter Cards)——遊戲操控的「反派」
| 類型 | 說明 | 關鍵屬性 |
|---|---|---|
| **敵人 Enemy** | 攻擊調查員 | 戰鬥值 Fight、生命 Health、閃避值 Evade、傷害/恐懼輸出、**關鍵字** |
| **詭計 Treachery** | 抽到即結算的負面事件 | 效果文字、可能引發檢定 |
| **地點 Location** | 構成「地圖」,可移動與調查 | 遮蔽值 Shroud、線索 Clues、連接符號、正/反面 |
| **密謀 Agenda** | 反派的計時器,累積「毀滅 Doom」後推進 | 毀滅門檻、推進效果 |
| **幕 Act** | 玩家的進度軸,累積「線索」後推進 | 線索門檻、推進效果 |
| **劇本卡 Scenario** | 該場的設置、特殊規則、混沌標記效果 | 設置流程、勝利條件、結局分支 |

### 1.4 標記與資源(Tokens)
線索 Clue、毀滅 Doom、傷害 Damage、恐懼 Horror、資源 Resource、充能/彈藥(Charges/Ammo)等,以及**混沌袋**內的混沌標記(見 §4)。

> **🔧 建模重點:** 所有卡牌共用一個「卡牌實體(CardInstance)」抽象,持有:靜態定義(CardDefinition)、動態狀態(位置、標記數、耗竭與否、附加的修正/附屬卡)。位置(Zone)包含:牌組、手牌、檯面、棄牌堆、被消滅、遭遇牌堆、威脅區(engaged)、地點上、混沌袋等。

## 2. 場上區域與地圖

- **地點(Locations)** 以連接符號構成地圖;調查員站在某地點,可沿連接**移動**。
- 地點有 **未揭示/已揭示** 兩面;首次進入時揭示,放上初始線索(依人數縮放,常見 = 每位調查員 1 個)。
- 每位調查員有自己的 **檯面(play area)**:支援卡、資源池、傷害/恐懼、被威脅(engaged)的敵人。
- **幕/密謀** 各自成一疊,並排推進。

> **🔧 建模重點:** 地圖是一張「圖(graph)」;移動、獵人(Hunter)敵人尋路、效果的「相鄰(connected)」判定都需要圖結構。線索/敵人歸屬於「地點」或「調查員」需明確。

## 3. 回合結構(Game Round)

每一輪固定四階段,循環直到劇本結束:

### 3.1 神話階段(Mythos)—— **第一輪跳過**
1. 在當前**密謀(agenda)**放上 1 個毀滅。
2. 比對場上總毀滅是否達密謀門檻 → 推進密謀(通常是壞事;推進時移除**場上所有**毀滅,含敵人/地點上的)。
3. 依 player order,**每位調查員各抽 1 張遭遇卡並結算**(敵人進入威脅區、詭計立即生效等)。

### 3.2 調查階段(Investigation)——遊戲主體
- 玩家**自選順序**(合作遊戲:團隊決定誰先行動、不強制固定順序),輪到的調查員可執行 **3 個行動**。可用行動:
  - **抽 1 張牌 / 獲得 1 資源 / 移動 / 打出卡牌 / 啟動能力**
  - **調查(Investigate)**:對所在地點做智力檢定 vs 遮蔽值,成功取得 1 線索。
  - **戰鬥(Fight)**:對敵人做戰鬥檢定 vs 敵人戰鬥值,成功造成傷害。
  - **閃避(Evade)**:對敵人做敏捷檢定 vs 敵人閃避值,成功使其耗竭並脫離交戰。
  - **交涉(Parley)/ 抽身(Resign)/ 交戰(Engage)** 等特定行動。
- 許多卡牌能力也需消耗行動或有「每回合一次」限制。

### 3.3 敵人階段(Enemy)
1. **獵人(Hunter)** 關鍵字敵人朝最近調查員移動 1 步。
2. 每個與調查員**交戰中**的敵人發動 1 次攻擊(造成傷害/恐懼)。

### 3.4 整備階段(Upkeep)
1. 重置行動數;將耗竭(exhausted)的卡/調查員回正(ready)。
2. **每位調查員抽 1 張牌、獲得 1 資源**。
3. 手牌上限檢查(通常 8 張,超過棄置)。

> **🔧 建模重點:** 回合是一台「階段狀態機」;每個階段/步驟切換都會開啟**能力視窗(ability window)**,讓觸發型能力有機會反應(見 §6)。「行動」是一種需要開銷與合法性驗證的操作;某些行動會引發**趁隙攻擊(Attack of Opportunity)**。

## 4. 技能檢定(Skill Test)與混沌袋(Chaos Bag)

**這是全遊戲的核心機制,也是引擎最需要精確實作的子系統。**

檢定流程:
1. **宣告檢定**:確定使用的技能(意/智/戰/敏)與難度(通常 = 敵人數值或地點遮蔽值)。
2. **投入技能卡(Commit)**:玩家從手牌投入圖示相符的卡,累加技能值(⚡wild 圖示百搭)。可多位調查員協助投入。
3. **抽混沌標記**:從混沌袋隨機抽 1 個標記,套用其修正。
4. **結算**:`(基礎技能 + 投入加值 + 卡牌/常駐修正 + 標記修正)` ≥ 難度 → **成功**,否則失敗。
5. **後續效果**:依成功/失敗、以及超出量結算(如成功造成傷害 = 武器值)。

### 混沌標記(Chaos Tokens)
- **數值標記**:`+1, 0, −1, −2, −3, −4, −5, −6, −8`(依難度不同組合)。
- **符號標記**:骷髏 💀、邪教徒 🌜、石板 📜、遠古種 🐙、**自動失敗 ⊗(觸手)**、**古老印記 ⭐**。
  - 符號標記的效果由**該劇本 + 難度**定義(💀/🌜/📜/🐙 通常隨劇本文字結算,可能造成額外壞事)。
  - ⭐ 的效果由**調查員**定義(通常是好事)。
  - ⊗ 幾乎必為自動失敗。
- **難度(Difficulty)**:簡單/標準/困難/專家,對應不同的標記集合(愈難負值愈多)。
- 進階機制:**封印(Seal)**(把標記暫時鎖在卡上)、**返還/加抽(return/draw again)**、依 token 觸發能力等。

> **🔧 建模重點:** 混沌袋是一個「多重集合 + 加權隨機抽取」服務,需支援封印/返還/連抽/「抽到 X 則…」的鉤子。**RNG 必須在權威伺服器端、且可設種子(seedable)以支援重播與除錯**。檢定本身是一台小狀態機(宣告→投入→抽標記→套修正→判定→後續),每一步都可能被卡牌能力介入。

## 5. 戰鬥、調查、敵人細節

### 5.1 交戰(Engagement)與攻擊
- 敵人進場通常「交戰」某位調查員(進入其威脅區)。
- 與敵人交戰時執行某些行動(移動、調查等)會引發**趁隙攻擊**。
- **戰鬥**:成功→造成傷害(= 你的武器/基礎傷害);失敗某些情況反被打。
- **閃避**:成功→敵人耗竭並脫離交戰;敵人耗竭時不會攻擊。

### 5.2 敵人關鍵字(部分)
獵人 Hunter(會移動追人)、冷漠 Aloof(不主動交戰)、龐大 Massive(與所有同地點者交戰)、反擊 Retaliate(你戰鬥失敗會反傷)、警覺 Alert(你閃避失敗會反傷)、精英 Elite、獵物 Prey/巡邏 Patrol(指定交戰目標規則)等。

### 5.3 調查與線索
- **調查**:智力 vs 遮蔽值,成功取得該地點 1 線索。
- 線索累積到「幕」門檻即可推進劇情;線索通常歸屬到調查員身上,再用於推進幕。

### 5.4 被擊敗(Defeat)
- 傷害 ≥ 生命 或 恐懼 ≥ 理智 → 調查員被擊敗(該劇本退場)。
- 敵人生命歸零 → 被擊敗(消滅)。

> **🔧 建模重點:** 關鍵字是一組可組合的「行為修飾」;引擎需要一個關鍵字/能力註冊表。傷害/恐懼的指派要處理「可吸收到支援卡(如盟友)」「分配」等規則。

## 6. 能力、時機與優先權(電子化最難的部分)

AHLCG 有一套類似 MtG 的**時機結構**,是完整規則引擎的靈魂:

- **能力類型**:
  - **常駐(Constant)**:持續改變遊戲狀態(如「你的戰鬥 +1」)。
  - **強制(Forced)**:條件滿足時**必須**觸發。
  - **反應/自由觸發(Reaction / Free triggered, `↺`)**:在特定「視窗」由玩家選擇是否觸發。
  - **啟動(Activate)**:需花行動的能力。
- **能力視窗(Ability Windows)**:遊戲流程在許多節點會「暫停」,依固定順序詢問各方是否要觸發反應能力(類似優先權傳遞)。
- **結算與巢狀**:一個效果可能引發另一個效果,需要**效果堆疊/巢狀結算**與明確的先後規則。
- **修正層級(Modifier layers)**:多個常駐修正同時作用時的計算順序(加值、設定值、上下限)。

> **🔧 建模重點:** 這一層決定引擎「像不像真的規則」。需要:①事件匯流排(每個遊戲動作發布 pre/post 事件);②觸發能力的登記與排程;③「請求玩家決策」的協定(引擎暫停,向對應客戶端要求選擇);④修正計算引擎。**這是整個專案技術風險最高、最該先做原型的部分。**

## 7. 戰役系統(Campaign)

- 多個劇本按順序遊玩,串成一段戰役。
- **戰役日誌(Campaign Log)**:記錄抉擇、擊殺、獲得的道具、密語等,影響後續劇本設置與分支。
- **經驗值(XP)**:劇本結束依「勝利點(Victory)」等結算,用於**升級牌組**(購入高等卡、替換卡)。
- **創傷(Trauma)**:被擊敗會累積肉體/精神創傷,跨劇本延續(降低生命/理智上限,嚴重則調查員陣亡/退場)。
- **結局分支(Resolution)**:劇本可能有多種結局,導向不同後續。

> **🔧 建模重點:** 戰役是一份**長期持久化的狀態文件**(調查員成長、日誌、庫存、分支旗標)。多人共享同一場戰役 → 存檔位置與同步是設計重點(見 [03](03-tech-requirements.md) 資料儲存)。牌組升級需要一套「牌組構築規則驗證器」。

---

## 8. 牌組構築(Deckbuilding)

- 由調查員定義:可用職業與等級、牌組張數(常見 30 張,不含弱點)、必帶招牌卡與招牌弱點、特殊構築規則。
- 卡片有等級 0–5,含高等卡需付 XP。
- 需驗證:張數、每卡上限(通常同名 ≤2)、職業/等級合法性、招牌/弱點強制納入、特殊限制。

> **🔧 建模重點:** 牌組構築是一個**規則驗證引擎**(輸入牌組 + 調查員 → 合法/違規原因)。與商業版的「擁有內容(entitlement)」檢查結合(只能用你擁有的卡)。

## 9. 電子化時的規則抽象總表

| 規則概念 | 程式抽象 |
|---|---|
| 卡牌 | `CardDefinition`(靜態)+ `CardInstance`(動態狀態) |
| 位置 | `Zone`(牌組/手牌/檯面/棄牌/消滅/威脅區/地點…) |
| 地圖 | 地點的 `Graph`(連接、相鄰、尋路) |
| 回合 | 階段/步驟 `StateMachine` |
| 行動 | `PlayerAction`(合法性驗證 + 開銷 + 趁隙攻擊判定) |
| 檢定 | `SkillTest` 子狀態機 |
| 混沌袋 | 加權隨機 + 封印/返還鉤子(**server 端 seedable RNG**) |
| 能力 | 事件匯流排 + 觸發登記 + 效果堆疊 + 修正層 |
| 決策 | 「引擎暫停→向客戶端請求選擇」的請求/回應協定 |
| 戰役 | 長期持久化狀態(日誌/XP/創傷/分支旗標) |
| 牌組構築 | 規則驗證引擎 + 擁有內容檢查 |

➡️ 這些抽象的系統化設計,見 [02-design-spec.md](02-design-spec.md)。
